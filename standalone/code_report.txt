1. PROJECT DESCRIPTION
--------------------------------------------------------------------------------
The 2048 Game is a single-player sliding tile puzzle game implemented as a 
fully self-contained HTML file. The objective is to combine numbered tiles on 
a customizable grid by sliding them in four cardinal directions, merging tiles 
of equal value to create progressively larger numbers, with the ultimate goal 
of creating a tile with the value 2048. This implementation consolidates the 
original multi-file JavaScript architecture into a single standalone HTML 
document with embedded CSS and JavaScript for maximum portability.

Version 2.0 introduces dynamic grid sizing (3×3 to 10×10, including non-square 
grids), unlimited undo functionality with keyboard shortcuts, and dimension-
specific game state persistence.

2. CORE SYSTEM ARCHITECTURE (UPDATED)
--------------------------------------------------------------------------------
A. Game Logic Engine:
   - Dynamic grid-based state management using 2D array structure (variable dimensions).
   - Tile objects track position (x, y), value, and merge history.
   - Move validation using vector-based traversal system adaptable to any grid size.
   - Win condition: Creating a 2048 tile (regardless of grid size).
   - Loss condition: No available moves (grid full + no adjacent matching tiles).

B. Input Management System:
   - Multi-modal input support:
     * Keyboard: Arrow keys (↑↓←→), WASD, Vim keys (HJKL)
     * Touch: Swipe gestures with 10px minimum threshold
     * Mouse: Click-based button controls
     * Undo: Ctrl+Z (Windows/Linux) or Cmd+Z (macOS)
   - Event-driven architecture with custom event emitter pattern.
   - Single KeyboardInputManager instance reused across grid size changes.

C. Rendering & Animation (HTML5 CSS Animations + Dynamic Styling):
   - Dynamic CSS generation for tile positioning based on grid dimensions.
   - Adaptive cell sizing algorithm maintains playability across all grid sizes.
   - CSS-based tile positioning using transform: translate() with calculated offsets.
   - Keyframe animations for tile appearance (@keyframes appear).
   - Merge animations with scale effect (@keyframes pop).
   - Score addition floating animations (@keyframes move-up).
   - RequestAnimationFrame for smooth visual updates.

D. Persistence Layer:
   - Dimension-specific LocalStorage keys (e.g., gameState_4x4, bestScore_5x5).
   - Automatic game state saving after each valid move per grid configuration.
   - Best score tracking isolated per grid dimension.
   - State validation prevents loading incompatible saved games.
   - Fallback to in-memory storage (fakeStorage) for unsupported browsers.

E. Undo System (NEW):
   - State snapshot captured before each valid move.
   - Stack-based architecture with 50-move history limit.
   - Full state serialization includes grid, score, and game flags.
   - Invalid moves (no tile displacement) do not consume undo stack space.

5. GAME MECHANICS & ALGORITHMS (UPDATED)
--------------------------------------------------------------------------------
A. Tile Movement Algorithm:
   - Vector-based directional mapping (0=Up, 1=Right, 2=Down, 3=Left).
   - Adaptive traversal order: Dynamically adjusts for width and height.
   - Traversal reversal based on movement direction prevents double-processing.
   - Farthest position calculation using iterative cell checking.
   - Merge detection: Adjacent tiles with equal values and no prior merge flag.

B. Score Calculation:
   - Points awarded equal to the value of newly merged tile.
   - Example: Merging two 64 tiles awards 128 points (2×64=128).
   - Cumulative scoring system with dimension-specific best score tracking.
   - Score persistence tied to grid configuration.

C. Random Tile Generation:
   - 90% probability: Generate tile with value 2.
   - 10% probability: Generate tile with value 4.
   - Spawns in random available (empty) cells after valid moves.

D. Grid Sizing Logic:
   - Cell dimension formula: (500 - 30 - (max(w,h) - 1) × 15) / max(w,h).
   - Container dimensions: (cellSize × w) + ((w-1) × 15) + 30 for width.
   - Dynamic CSS generation: Position classes for every (x, y) coordinate pair.
   - Font size scaling: Inversely proportional to cell size for readability.

E. Undo State Management:
   - Snapshot timing: Captured immediately before tile movement logic.
   - Restoration process: Deserialize grid from stack, rebuild Tile objects.
   - Stack pruning: FIFO removal when exceeding 50-move limit.
   - Rollback trigger: Pop most recent state and re-actuate display.

16. LESSONS LEARNED FROM VERSION 2.0 DEVELOPMENT
--------------------------------------------------------------------------------
- Dynamic CSS Generation Trade-offs: Runtime style injection adds flexibility 
  but increases complexity. The one-time performance cost (~5-10ms) is acceptable 
  for grid size changes, but would not scale for real-time updates.
  
- LocalStorage Key Namespacing: Dimension-specific keys prevent cross-contamination 
  but create potential for orphaned data. Future versions should implement 
  storage cleanup for unused grid configurations.
  
- Undo Stack Memory Management: Initial unbounded stack caused memory issues on 
  extended play sessions (500+ moves). The 50-move cap balances functionality 
  with resource constraints (empirically tested).
  
- Grid Initialization Timing: Calling updateGridSize() on every actuate was 
  necessary to ensure grid visibility but creates redundant DOM operations. 
  Performance impact is negligible for grids ≤7×7 but noticeable at 10×10.
  
- State Validation Criticality: Version 2.0 initially suffered from dimension 
  mismatch bugs when loading saved states. Strict validation (checking 
  previousState.grid.width/height) was essential to stability.

20. ARCHITECTURAL DECISIONS (NEW SECTION)
--------------------------------------------------------------------------------
A. Single Actuator vs. Multiple Instances:
   - Decision: Reuse HTMLActuator instance across grid changes.
   - Rationale: DOM element references remain valid, reduces initialization overhead.
   - Trade-off: Requires manual grid rebuild on dimension change.

B. Dimension-Specific Storage Keys:
   - Decision: Separate localStorage keys per grid configuration.
   - Rationale: Prevents state corruption, allows parallel gameplay across sizes.
   - Trade-off: Increased storage footprint, potential orphaned data.

C. Undo Stack Size Limit:
   - Decision: Cap at 50 moves via FIFO pruning.
   - Rationale: Balance between functionality and memory usage (~6KB per 50 moves).
   - Trade-off: Cannot undo beyond 50th previous move (acceptable for casual play).

D. CSS Generation Approach:
   - Decision: Runtime style injection vs. pre-generated classes.
   - Rationale: Supports arbitrary grid dimensions without code bloat.
   - Trade-off: One-time performance cost on grid size change (~5-10ms).

E. Grid Rebuild Frequency:
   - Decision: updateGridSize() called on every actuate.
   - Rationale: Guarantees grid visibility and correct positioning.
   - Trade-off: Redundant DOM operations (mitigated by modern browser optimization).
